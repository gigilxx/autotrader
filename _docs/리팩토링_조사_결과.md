### ① `get_account()` 이중 호출 — 진입마다 0.5초 추가 지연

**파일**: `autotrader/engine.py:225`, `autotrader/execution.py:54`

**문제**

진입 신호가 발생하면 `_watch_entry()` 단일 흐름에서 `get_account()`가 두 번 호출된다.

```python
# engine.py:225 — 사이징용 현금 조회
available_cash = self.router.broker.get_account().cash

# execution.py:54 — OrderRouter.place() 내부 리스크 게이트용 잔고 조회
account = self.broker.get_account()
```

KIS rate limit으로 인해 `_throttle`이 두 번째 호출 전 최소 0.5초(모의) 대기를 강제한다. 즉 진입 신호마다 불필요한 0.5초 지연이 추가된다. `_force_entry()`도 동일한 패턴이다(`engine.py:358`).

**수정 방법**

`_watch_entry()`에서 조회한 `account`를 `OrderRouter.place()`에 전달하거나, `available_cash`를 파라미터로 전달한다.

```python
# engine.py _watch_entry()
account = self.router.broker.get_account()
available_cash = account.cash

# router.place()에 전달 (시그니처 변경)
d = self.router.place(order, prefetched_account=account)

# execution.py OrderRouter.place()
def place(self, order: OrderRequest, prefetched_account=None) -> Decision:
    account = prefetched_account or self.broker.get_account()
    ...
```

```python
def load_daily_state(self, today):
```

### ③ `_conn()/_db()` DB 연결 코드 3중 복사 + WAL 설정 불일치

**파일**: `autotrader/state.py:79`, `ui/api/main.py:58`, `autotrader/telegram_control.py:74`

**문제**

SQLite 연결 컨텍스트 매니저가 세 파일에 독립적으로 구현되어 있다. 더 큰 문제는 WAL 설정이 제각각이다.

| 파일 | WAL 설정 |
|---|---|
| `state.py` | ❌ 없음 |
| `api/main.py` | ✅ `PRAGMA journal_mode=WAL` |
| `telegram_control.py` | ✅ `PRAGMA journal_mode=WAL` |

봇 프로세스(`state.py`)가 WAL 없이 DB를 먼저 생성하면 DELETE 모드로 초기화되어 멀티프로세스 동시 접근 시 쓰기 잠금 충돌 가능성이 있다.

**수정 방법**

`state.py`의 `_init_db()` 또는 `_conn()`에 WAL을 추가하는 것만으로 우선 해결한다. 장기적으로는 `telegram_control.py`가 직접 SQLite를 열지 않고 `StateManager`를 임포트해 사용하도록 전환한다.

```python
# state.py _init_db() 첫 줄 추가
def _init_db(self) -> None:
    with self._conn() as cx:
        cx.execute("PRAGMA journal_mode=WAL")
        cx.executescript("CREATE TABLE IF NOT EXISTS ...")
```

```python
# telegram_control.py — StateManager 재사용으로 전환
from autotrader.state import StateManager

_sm = StateManager(Path(os.getenv("STATE_DB", "state.db")))

def _set_flag(key, value="1"):
    _sm.set_control_flag(key, value)

def _clear_flag(key):
    _sm.clear_control_flag(key)
```

### ⑥ `config.py` 필드 타입이 `None`으로만 선언

**파일**: `autotrader/config.py:58-60`

**문제**

```python
@dataclass
class AppConfig:
    risk: RiskConfig = None      # Optional[RiskConfig]로 선언해야 하지만
    cost: CostConfig = None      # 정적 분석기는 None으로 추론
    strategy: StrategyConfig = None
```

`__post_init__`에서 실제 값을 채우지만, mypy/pyright는 `cfg.risk`를 `None`으로 추론해 `cfg.risk.stop_loss_pct` 접근에서 경고를 발생시키지 않는다. 즉 타입 체커가 작동하지 않는다.

**수정 방법**

```python
from dataclasses import dataclass, field

@dataclass
class AppConfig:
    risk: RiskConfig = field(default_factory=RiskConfig)
    cost: CostConfig = field(default_factory=CostConfig)
    strategy: StrategyConfig = field(default_factory=StrategyConfig)
    # __post_init__에서 환경변수 오버라이드 유지
```

**파일**: `autotrader/engine.py:212-280`

**문제**

```
_watch_entry()
├── 진입 조건 체크 (시장 필터, 시간, 목표가)
├── 포지션 사이징
```

```python
def _confirm_fill(self, odno: str, sym: str, requested_qty: int, px: int) -> tuple[int, int]:
    """주문 체결을 확인하고 (filled_qty, fill_price)를 반환한다."""
    try:
        fill = self.router.broker.wait_for_fill(odno, sym, requested_qty)
        ...
        return fill.filled_qty, int(fill.avg_price) or px
    except Exception as e:
        logger.warning("체결 조회 실패 %s: %s — 요청 수량으로 가정", sym, e)
        return requested_qty, px
```

```python
sorted_bars = sorted(bars, key=lambda b: b.date)  # O(N log N) — 불필요
```

```python
closes = [b.close for b in bars if b.close > 0]

ma = sum(closes[:usable]) / usable
```
